# Output Format Standard

Risk-triggered output requirements for governance skills. Not every skill
invocation must produce a full structured record — the depth scales with risk.

## Format by Severity

### L0/L1 — Minimal

One line in commentary is sufficient. Example:

    [L1] Added pytest-cov to dev dependencies. Reversible via uv remove.

### L2 — Compact

Required fields:

- `affected_resources`: list of files, services, databases, or endpoints touched
- `approval_required`: yes/no, and from whom if yes
- `rollback/verification`: how to undo or how to confirm correctness

Example:

    [L2] catalog-api: add sku_dedup endpoint
    affected_resources: catalog-api/routes.py, catalog-api/schema.py
    approval_required: no
    rollback/verification: revert commit; verify with pytest tests/api/test_sku.py

### L3/L4 — Full

Required fields:

- `risk_class`: L3 or L4
- `affected_resources`: complete list
- `approval_required`: yes, and from whom
- `rollback_plan`: step-by-step undo or compensation
- `evidence`: what verification was performed, with command output or link
- `decision`: proceed / blocked / escalated
- `skill`: which skill produced this record

Example:

    [L3] support workflow: modify ticket-classification prompt
    affected_resources: workflow version "ticket-structure-v3", prompt config
    approval_required: yes, manual review of prompt diff
    rollback_plan: revert to workflow version v2 (saved as draft)
    evidence: evaluation suite V7 N=60, 9/9 gates pass, results at tests/eval/v7-run-2026-07-08.json
    decision: proceed
    skill: data-contract-and-lineage

## Placement

Records live in the task's natural output channel:

- L0/L1: commentary channel during task execution
- L2: commentary or final answer, as appropriate
- L3/L4: final answer, and optionally persisted to a project audit note if the
  project has one

No global governance-trail file is generated by default. Conversation history
is not a durable fact source; use the task's owning Git history, README, ADR,
RUNBOOK, or existing audit store when a persistent record is required.

## Field Definitions

- `risk_class`: L0-L4 per severity-matrix.md
- `affected_resources`: files, services, databases, endpoints, config keys,
  containers, or cloud resources
- `approval_required`: boolean, plus role or person
- `rollback_plan`: concrete steps, not "revert commit"; name the command or
  procedure
- `evidence`: must satisfy the AGENTS completion gate; link to fresh test,
  command, API, or evaluation output
- `decision`: one of `proceed`, `blocked`, `escalated`
- `skill`: the skill name that produced this record
